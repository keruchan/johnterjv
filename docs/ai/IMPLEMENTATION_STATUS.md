Status: Registration, profile, authentication, four-role page access, database-backed exceptional permissions, Superadmin user-management flows, notification creation, general audit logging, the Tree Cutting Permit backend foundation, Community draft/submission/list/detail/document intake, RPS online/original document review, site-inspection/tree-verification, RPS review/approval/decline/return, immutable policy-versioned donation requirements, EMS receipt/physical verification, final RPS donation confirmation, signed-permit preparation/release with a stored 50-90 day validity, signed-permit scan upload/protected download, automatic expiration processing, and cutting-completion recording are verified. The complete Phase 2 permit lifecycle (submission through completion or expiration) is now implemented; the validity-start rule and several office policies remain unverified business decisions. Signed-permit scan upload/download (with optional signing personnel/date and recipient) is implemented. Project-owner decision: the official permit is a physical wet-ink document claimed in person at the CENRO office and is not generated or delivered digitally; the system communicates pickup status to the applicant, so automatic permit-document generation is intentionally out of scope. The seedling inventory and public seedling-request program (Phase 3) is now implemented independently of the permit workflow, service- (53/53) and HTTP-verified. The illegal-logging incident report program (Phase 3) is now implemented independently of the permit and seedling workflows, service- (45/45) and HTTP-verified, introducing a new `illegal_logging_processing` exceptional permission for Superadmin enforcement access. Area Management (Phase 3) is now implemented as a CENRO-internal reference registry, service- (25/25) and HTTP-verified, deliberately kept separate from the permit workflow. Public Advisories (Phase 3) is now implemented as a CENRO-authored, Community-visible announcement feed with a draft/published/archived lifecycle, service- (33/33) and HTTP-verified, independent of every other module. Analytics and reporting (Phase 3) is now implemented as a deep read-only analytics layer with descriptive, predictive, and prescriptive tiers: a CENRO cross-domain overview (RPS/Superadmin) and an EMS Inventory Reports view (EMS-only), each date-filterable with a CSV export, service- (41/41 aggregates + 37/37 deep-analytics math) and HTTP-verified, adding no tables, audit categories, or notification types. Visualizations use Chart.js 4 (CDN) themed to the dashboard palette; forecasting is transparent ordinary least-squares regression (no opaque ML).

Status meaning:

- **Implemented**: backend behavior exists in source; this does not imply browser/end-to-end verification.
- **Partially implemented**: some functional backend or protected page exists, but the module is incomplete.
- **Planned**: current UI advertises or links to the module without a backend workflow.
- **Unverified**: appears only in the external domain brief, with no current repository artifact.

## Engineering Limitations

- The database contract uses PDO's MySQL driver, while the verified local server is MariaDB 10.4.32.
- Community permit UI routes and transactional scan/original-document/inspection/decision/donation-requirement/EMS-receipt services exist, including RPS, specifically permitted Superadmin, owner read-only, and EMS mutation surfaces; final RPS donation confirmation, signed-permit release, validity/expiration, and cutting-completion pages are now implemented as well (`includes/permit_release.php` with its CENRO action pages).
- Database credentials and starter account credentials are stored in source/setup files.
- There is no full shared PHP page layout, migration runner, or package manifest. Protected pages share the role-aware navigation component; focused PHP and local HTTP permit validation scripts now exist.
- `ajax/`, `assets/`, `js/`, and public `uploads/` contain no implementation files; sensitive permit scans and inspection photographs use separate configurable private directories outside the web root.

## Authentication TODOs

- Login throttling/rate limiting and failed-login auditing are implemented: `includes/auth.php` enforces independent sliding-window counters (5 failures per identifier, 20 per IP, 15-minute window, all configurable) backed by `tbl_login_attempts`, which records every attempt (success or failure, with responsible user when known) and doubles as the failed-login audit trail. Neither counter mutates account status, so a mistyped-password lockout self-heals after the window elapses.
- Add an operational grant/revoke UI and policy for exceptional permissions if database-level provisioning is not sufficient.
- Password reset is implemented per project-owner confirmed scope: admin-initiated reset only, via a "Reset password" action in User Management. No self-service email flow exists, so email verification/sending infrastructure remains out of scope.
- Idle (30 min default) and absolute (480 min default) session timeouts are implemented in `includes/auth.php`, enforced on every `require_roles()`/`require_role()` call; both are configurable via `CERTREEFY_SESSION_IDLE_TIMEOUT_MINUTES`/`CERTREEFY_SESSION_ABSOLUTE_TIMEOUT_MINUTES`. An expired session is destroyed and redirected to a login page message explaining which limit was reached; a pre-existing session with no tracking timestamps is treated as freshly started rather than immediately expired.
- Logout is now a CSRF-protected POST action (HTTP-verified: GET returns 405, a mismatched token returns 403 without ending the session, and every logout button across all role dashboards/pages posts a session-scoped token generated in `require_roles()`).
- An audit-history viewer is implemented (`pages/cenro/audit-history.php`, Superadmin-only): filterable/paginated `tbl_audit_trail` and `tbl_login_attempts` registries with independent, cross-preserved filters. No retention/purge policy exists yet; rows accumulate indefinitely pending an office decision on retention duration.
